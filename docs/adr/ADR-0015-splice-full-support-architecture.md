# ADR-0015: Splice Full-Support Architecture and Boundaries

**Status:** Proposed
**Date:** 2026-04-02
**Authors:** Charles Dusek

## Context

cantonctl currently has two local runtime models:

- `cantonctl dev` starts a single-participant sandbox via `src/lib/dev-server.ts`.
- `cantonctl dev --full` starts a Canton-only Docker topology via `src/lib/dev-server-full.ts` and `src/lib/topology.ts`.

Those flows share the same core assumptions:

- `src/lib/config.ts` models networks as a single target with `type: sandbox | remote | docker`.
- `src/lib/ledger-client.ts` speaks only Canton JSON Ledger API V2.
- `src/lib/deployer.ts` uploads DARs to a single JSON Ledger API target, or to every participant in a generated Canton topology.
- `src/lib/serve.ts`, `src/commands/status.ts`, and `src/commands/playground.ts` treat the world as either one sandbox or a generated multi-participant Canton topology.
- `src/lib/scaffold.ts` generates only a sandbox-style `local` network.

That is not equivalent to Splice LocalNet.

`dev --full` today is a generated Canton topology with:

- one Canton container,
- a synchronizer,
- multiple participants,
- JSON Ledger API endpoints,
- sandbox-style JWT auth,
- DAR upload fan-out across participants.

Splice LocalNet adds additional application and protocol surfaces that are outside that model:

- validator application endpoints and validator-specific party routing,
- Scan or other indexed read services,
- wallet gateway / CIP-0103 browser flows,
- OIDC-backed authentication and account bootstrap,
- Splice Token Standard APIs and release coupling,
- upstream OpenAPI and OpenRPC contracts that cantonctl does not currently own.

If cantonctl treats current `dev --full` as "Splice support", it will blur two separate concerns:

1. Canton runtime orchestration, which cantonctl already owns.
2. Splice application stack orchestration, which should remain anchored to upstream artifacts and contracts.

The architecture must add full Canton plus Splice support without regressing the existing sandbox-first workflow from [ADR-0004](0004-sandbox-first-local-dev.md), the delegation boundary from [ADR-0010](0010-hybrid-architecture.md), or the current Docker topology from [ADR-0014](0014-dev-full-multi-node-topology.md).

## Decision

cantonctl will model "full Canton plus Splice support" as explicit runtime profiles with strict capability boundaries and upstream delegation.

### 1. Keep current local runtimes intact

The existing runtimes remain first-class and stable:

- `sandbox` profile: current `cantonctl dev`
- `canton-full` profile: current `cantonctl dev --full`

`--full` remains a compatibility alias for `--profile canton-full` when the profile architecture lands.

### 2. Treat Splice LocalNet as a distinct profile, not an extension of `canton-full`

The new target runtime is `splice-localnet`, separate from `canton-full`.

Reason:

- `canton-full` is a Canton-only topology generated by cantonctl.
- `splice-localnet` is an application stack with additional services, contracts, auth flows, and upstream release semantics.

This avoids overloading `dev --full` with behavior that belongs to a different runtime class.

### 3. Separate profiles from networks

The future config model distinguishes:

- `profiles`: local runtime bundles and their capabilities
- `networks`: deploy/status/auth targets

Profiles describe how a stack is started locally. Networks describe where a command is pointed.

This separation is required because a single profile can expose multiple service endpoints, while the current `networks.*.url` model only captures one JSON API target.

### 4. Adopt a capability model for command integration

Commands should resolve named capabilities instead of assuming a single ledger URL.

Target capability set:

| Capability | Purpose | Initial owner |
|------------|---------|---------------|
| `ledger-json-api` | DAR upload, contracts, commands, parties | cantonctl |
| `participant-admin` | participant diagnostics and topology checks | cantonctl |
| `validator-api` | validator-facing application APIs | upstream Splice |
| `scan-api` | indexed reads and explorer data | upstream Splice |
| `wallet-openrpc` | wallet / CIP-0103 integration | upstream Splice |
| `oidc` | auth issuer and login bootstrap | upstream Splice |

`src/lib/ledger-client.ts` remains the stable adapter for `ledger-json-api`. New Splice-facing adapters should wrap upstream contracts, not invent them.

### 5. Stable vs experimental boundaries

Stable:

- `sandbox` runtime
- `canton-full` runtime
- JSON Ledger API based build, deploy, status, serve, and playground flows
- config and command compatibility for existing users
- profile resolution and capability discovery infrastructure

Experimental:

- `splice-localnet` runtime orchestration
- validator, Scan, wallet, and OIDC service adapters
- serve/playground integration against wallet-backed Splice flows
- Token Standard convenience helpers layered on top of upstream APIs

Experimental features must be explicitly labeled and must not weaken existing sandbox or `canton-full` behavior.

### 6. Ownership boundary: what cantonctl owns vs delegates upstream

cantonctl owns:

- command UX, profile selection, and config resolution,
- lifecycle orchestration for local stacks,
- health checks, status reporting, and doctor diagnostics,
- generated local overlay files and environment wiring,
- scaffold defaults and project-level configuration,
- thin adapters around stable upstream APIs.

cantonctl does not own:

- Splice business logic or reference application semantics,
- validator, Scan, wallet gateway, or OIDC server implementations,
- upstream OpenAPI / OpenRPC specifications,
- Token Standard protocol rules,
- bespoke replacements for upstream launchers, images, or bootstrap scripts when upstream artifacts already exist.

The rule from ADR-0010 still applies: cantonctl orchestrates and integrates; it does not reimplement upstream runtimes.

### 7. Target command architecture

Target command surface:

| Command | Target architecture |
|---------|---------------------|
| `dev` | `cantonctl dev --profile sandbox|canton-full|splice-localnet` |
| `status` | Capability-aware status for a selected profile or network |
| `doctor` | Profile-aware prerequisite checks, including Splice bundle prerequisites where applicable |
| `serve` | Stable for `sandbox` and `canton-full`; `splice-localnet` remains experimental until wallet and validator routing contracts are proven |
| `playground` | Stable for ledger-centric profiles first; Splice wallet-backed UX follows later |
| `deploy` | Continues to support DAR deployment to ledger-capable targets; validator-specific or app-specific deployment remains delegated upstream until a stable upstream contract exists |

### 8. Target config/profile architecture

The target shape is:

```yaml
version: 1

project:
  name: my-app
  sdk-version: "3.4.11"

default-profile: sandbox

profiles:
  sandbox:
    kind: sandbox
    ports:
      canton: 5001
      ledger-json-api: 7575

  canton-full:
    kind: canton-full
    base-port: 10000
    canton-image: ghcr.io/digital-asset/decentralized-canton-sync/docker/canton:0.5.3

  splice-localnet:
    kind: splice-localnet
    experimental: true
    upstream:
      distribution: splice-localnet
      version: 0.5.x
    capabilities:
      ledger-json-api: auto
      validator-api: auto
      scan-api: auto
      wallet-openrpc: auto
      oidc: auto

networks:
  local:
    kind: ledger
    profile: sandbox

  localnet:
    kind: splice
    profile: splice-localnet
```

If this cannot be added without ambiguous behavior, cantonctl should introduce a new config version rather than overload `networks.type`.

## Consequences

**Positive:**

- Preserves the existing sandbox-first developer experience.
- Makes it explicit that `dev --full` is Canton realism, not full Splice parity.
- Allows Splice support to land incrementally without rewriting stable Canton code paths.
- Keeps cantonctl aligned with upstream release artifacts and protocol contracts.
- Creates a clean path for future command and scaffold evolution.

**Negative:**

- Adds a profile and capability layer that does not exist today.
- Introduces a temporary split between stable Canton support and experimental Splice support.
- Requires discipline to avoid leaking Splice-specific assumptions into generic ledger code.
- May require a config migration if the current schema cannot express profiles cleanly.
